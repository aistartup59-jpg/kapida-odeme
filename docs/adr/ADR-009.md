# ADR-009: YAGNI (You Aren't Gonna Need It)

- **Status:** Accepted
- **Date:** 2026-07-13

## Context

Epic 2 (Merchant Provider Management) introduced a `priority` field on the merchant payment provider API surface (create DTO, update DTO, response DTO, and service validation). The field anticipated future provider failover/routing, but no such capability exists today: the architecture enforces exactly one active provider per merchant, and `PaymentEngine` selects that single active provider directly. The field had no current functional use case — it was speculative surface added ahead of the feature that would justify it.

This is a recurring risk for a provider-independent, modular platform: it is easy to add fields, abstractions, or configuration ahead of need "because we'll want it eventually," which expands the API and domain model without a present business capability to justify it, and risks locking in a shape that doesn't fit the real feature once it's built.

## Decision

Kapıda Ödeme follows YAGNI (You Aren't Gonna Need It):

- Do not expose API fields for features that do not yet exist.
- Do not add speculative fields or abstractions.
- New capabilities must be introduced only when the corresponding business capability is implemented.
- Keep the domain model minimal.
- Future requirements must not shape today's API.

When a field or abstraction is proposed, it must trace to a business capability that exists in the current sprint. If it only supports a capability planned for later, it is deferred until that capability is actually implemented.

## Consequences

**Advantages:**

- Smaller, more honest API surface — clients cannot infer behavior (e.g. ranking/failover) that does not exist.
- Domain model stays aligned with implemented capability, not anticipated capability.
- Reduces the chance of locking in a shape that has to be reworked once the real feature is built.

**Disadvantages:**

- Fields removed under this rule may need to be reintroduced later, designed against the real requirements of the feature that needs them.
- Requires discipline during implementation to resist adding "obviously useful" fields ahead of need.

## Alternatives Considered

- Keep speculative fields in place "since they might be needed soon": rejected because it expands API surface and domain model without a present business capability, and risks the field's shape not matching the eventual real feature.
