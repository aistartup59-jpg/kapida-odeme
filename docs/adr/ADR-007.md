# ADR-007: Payment Orchestration Architecture

- **Status:** Accepted
- **Date:** 2026-07-11

## Context

Kapıda Ödeme must support multiple payment providers without coupling business logic to any specific provider. Merchants need to be able to switch providers independently. The system must be extensible without requiring changes to core business logic.

## Decision

Kapıda Ödeme is a payment orchestration platform.

Business logic must never reference a specific provider (ParamPOS, iyzico, PayTR, Sipay, or any other).

The call chain is:

```
PaymentService
      ↓
PaymentEngine
      ↓
PaymentProviderFactory
      ↓
PaymentProvider Interface
      ↓
Provider Adapter
      ↓
Provider API
```

Business logic communicates only through the `PaymentProvider` interface.

Each merchant selects their own provider. Merchant credentials belong to the merchant.

A provider implementation is responsible for:

- `createPayment`
- `generateBankQR`
- `createPaymentLink`
- `cancelPayment`
- `refundPayment`
- `getPaymentStatus`
- `handleWebhook`

Adding a new provider must require only:

- one new adapter
- provider registration

No business logic changes are allowed.

## Consequences

- Provider coupling is eliminated from business logic.
- New providers can be added without touching PaymentService or PaymentEngine.
- Merchant-level provider configuration becomes possible.
- Testing can use a mock adapter without any real provider dependency.

## Alternatives Considered

- Direct provider calls from PaymentService: rejected because it creates hard coupling and makes provider switching require widespread code changes.
- One shared provider for all merchants: rejected because merchants must be able to configure their own provider accounts.
