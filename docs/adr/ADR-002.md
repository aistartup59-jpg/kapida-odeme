# ADR-002: Hybrid and Partial Payments

- **Status:** Accepted
- **Date:** 2026-07-09

## Context

Merchants and employees need to accept payment across multiple methods and multiple attempts for a single order. The domain model must reflect this without ambiguity.

## Decision

A PaymentRequest may be completed using multiple successful Transactions.

PaymentRequest stores:

- `totalAmount`
- `paidAmount`

`remainingAmount` is never stored. It must always be calculated as:

```
remainingAmount = totalAmount - paidAmount
```

Payment status values:

- PENDING
- PARTIALLY_PAID
- PAID
- FAILED
- EXPIRED
- CANCELLED

Example:

- 500 TL QR + 250 TL CASH = 750 TL PAID

## Consequences

- No derived fields are stored, eliminating sync bugs.
- Status transitions are deterministic and driven by paidAmount vs totalAmount.
- Any combination of payment methods can settle a single order.

## Alternatives Considered

- Storing `remainingAmount`: rejected because it introduces a derived field that can fall out of sync.
- One payment method per order: rejected because the product requires hybrid payment support.
