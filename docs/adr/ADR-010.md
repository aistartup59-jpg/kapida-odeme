# ADR-010: Reuse Before Create

- **Status:** Accepted
- **Date:** 2026-07-14

## Context

As the project grows, creating parallel services, models, DTOs, interfaces, or modules increases architectural complexity, duplicates business logic, and violates YAGNI ([[ADR-009]]). Epic 3 surfaced a concrete instance of this risk: its literal requirements asked for a new `PaymentEngine` service and a new generic execution result, both of which already existed in substance (`PaymentEngineService`, `PaymentEngineResult`). Implementing it literally would have created a second, parallel orchestration path duplicating the existing one.

## Decision

- Before creating any new service, model, DTO, interface, entity, repository, or module, search the existing codebase.
- If an equivalent already exists, reuse, extend, or refactor it.
- Prefer refactoring over creating parallel abstractions.
- New abstractions require architectural justification.
- Duplicate orchestration paths are prohibited.
- Reuse existing architecture whenever possible.

## Consequences

- Smaller and more maintainable codebase.
- Lower technical debt.
- Better consistency.
- Easier onboarding.
- Reduced duplication.

## Alternatives Considered

- Implementing each epic's requirements literally, even when it duplicates existing services or models: rejected because it produces parallel orchestration paths and conflicting sources of truth as the codebase grows.
