# ADR-011: Payment Lifecycle Ownership

- **Status:** Accepted
- **Date:** 2026-07-14

## Context

Epic 4 found that `TransactionEngineService` validated status transitions via `PaymentStateMachineService.canTransition()` but then mutated `PaymentRequest.status` and persisted it itself. Validation was centralized; the actual write was not. As more services come to depend on `PaymentRequest` (transactions, refunds, cancellations, expiry), each one mutating `.status` directly would produce multiple, inconsistent write paths for the same field, undermining the state machine's purpose.

## Decision

Only `PaymentStateMachineService` may mutate `PaymentRequest.status`.

No other service may assign the `status` property directly.

Creating a new `PaymentRequest` with the initial `PENDING` state is allowed.

All subsequent lifecycle changes must go through `applyTransition()`.

## Consequences

- There is exactly one code path that can change a `PaymentRequest`'s status, so every transition is validated against the allowed-transition matrix.
- Future services (refunds, cancellations, expiry) must depend on `PaymentStateMachineService.applyTransition()` rather than writing to `.status` themselves.
- Initial creation is exempt because there is no prior state to transition from.

## Alternatives Considered

- Letting each consuming service validate via `canTransition()` and write `.status` itself: rejected because it already produced an inconsistency (`TransactionEngineService` writing `.status` directly) and does not scale as more services touch `PaymentRequest`.
