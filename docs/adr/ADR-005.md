# ADR-005: Single Payment Endpoint with Identity from JWT

- **Status:** Accepted
- **Date:** 2026-07-09

## Context

The API must expose payment request creation in a way that prevents clients from spoofing merchant or employee identity. A clean, single-endpoint design is preferred over separate merchant/employee routes.

## Decision

Use a single endpoint for payment request creation:

```
POST /payments
```

`merchantId` and `employeeId` must always be derived from the authenticated JWT.

They must never be accepted from the client request body.

Identity resolution rules:

- If the JWT `type` is `merchant`: `sub` is the merchant ID.
- If the JWT `type` is `employee`: `sub` is the employee ID; the merchant is resolved via the employee record.

## Consequences

- Clients cannot impersonate a different merchant or employee.
- The endpoint is unified for both merchant and employee callers.
- Service logic is responsible for identity resolution based on JWT claims.

## Alternatives Considered

- Separate endpoints per caller type: rejected to reduce API surface and avoid duplication.
- Accepting `merchantId` from the client: rejected because it allows identity spoofing.
